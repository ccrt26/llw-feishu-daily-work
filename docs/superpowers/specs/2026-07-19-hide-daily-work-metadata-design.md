# 工作日记内部元数据隐藏设计

## 目标

让 `亚信工作/每日工作/YYYY年MM月DD日/工作记录.md` 只包含用户可阅读的工作日记内容，即使打开 Markdown 源文件也不再出现 `llw-record-start`、`llw-record-data`、`llw-record-end` 或 Base64 编码。

同时保留以下现有能力：

- 同一北京时间自然日只有一个 Markdown 文件。
- 新记录按时间顺序排列。
- 自然语言补充能够更新原记录，不会按补充消息发送日另建记录。
- 飞书消息重复投递时不会重复入库。
- 整理内容和逐字原文都保留且明确区分。
- U 盘插入另一台已部署本地组件的 Mac 后，可以继续补充已有记录。

## 存放边界

采用“U 盘文档索引 + Mac 本地运行组件”的混合模式。

### U 盘

- 用户可读 Markdown：`亚信工作/每日工作/YYYY年MM月DD日/工作记录.md`。
- Skill 主版本：`.agents/skills/feishu-daily-work/`。
- 隐藏文档索引：`.llw-system/indexes/feishu-daily-work/records.json`。

隐藏索引属于知识库文档的结构化索引，不是运行日志或飞书事件状态。索引只保存随机记录编号、日期、排序字段、整理后的记录、原文及其先后关系。不得保存 App ID、App Secret、访问令牌、用户 ID、聊天 ID、飞书消息 ID或完整飞书事件。

### Mac 本地

- 程序、Node 运行时、CLI、LaunchAgent 和依赖。
- 飞书身份绑定、消息投递去重、未回复状态、会话上下文和心跳。
- 日志、迁移备份和恢复信息。
- 所有本地状态文件权限均为 `0600`，目录权限为 `0700`。

## 数据模型

新增独立的 `RecordStore`，负责读写隐藏文档索引。索引使用带版本号的 JSON：

- `version`：索引格式版本。
- `records`：以随机 `record_id` 为键的记录集合。
- 每条记录包含 `occurred_date`、时间、标题、人物、地点、整理内容、后续事项、排序键和按顺序排列的原文片段。
- 原文片段只标记 `initial` 或 `supplement`，不保存飞书消息标识。

记录编号使用本地安全随机值，不再由飞书消息 ID 推导。飞书投递去重继续由 Mac 本地 `StateStore` 负责。

`StateStore` 增加写入预留状态：在修改 U 盘前，先把本次飞书消息与已生成的随机记录编号、原文片段编号和目标操作保存在 Mac 本地。相同消息重试时必须复用预留编号，不得重新生成。预留状态只存在 Mac，不进入 U 盘索引。

索引写入采用临时文件、`fsync` 和同目录原子重命名，文件权限固定为 `0600`。每次读取均执行 Schema 级校验；版本未知、字段不合法或记录日期与目标文件不一致时拒绝写入 Markdown。

## 数据流

### 新建记录

1. AI + Skill 只负责判断 `create_record` 并返回受 Schema 约束的内容。
2. 服务生成随机记录编号和原文片段编号，并在 Mac 本地 `StateStore` 原子保存写入预留。
3. 相同消息重试时复用预留编号；确定性写入器生成排序键。
4. `RecordStore` 原子保存索引。
5. 写入器从索引重新渲染对应日期的纯净 Markdown。
6. 写回后逐字读取并核对预期内容，再把本地预留转为已提交结果并回复飞书。

### 补充记录

1. `RecordCatalog` 从隐藏索引生成最多 20 条候选，不向 AI 暴露原文、索引路径或飞书标识。
2. AI + Skill 选择唯一候选记录。
3. 服务为补充原文生成随机片段编号，并在 Mac 本地保存写入预留；重试时复用该编号。
4. 写入器在索引中追加补充原文并更新结构化内容，日期沿用目标记录日期。
5. 重新渲染目标日期 Markdown，不创建补充消息发送日文件。

### 故障恢复

Mac 本地写入预留先于索引提交，索引先于 Markdown 提交。若任一步后进程退出，同一消息重试时复用预留编号，并根据预留、索引和 Markdown 的实际状态完成剩余步骤，不创建第二条记录。服务启动时执行只读一致性检查；发现索引与 Markdown 不一致时停止自动入库并记录不含用户原文的错误代码，不静默覆盖用户文件。

## Markdown 输出

每日文件只保留以下内容：

- 日期一级标题。
- 每条记录的独立二级标题。
- 关键信息 callout。
- “整理后记录”。
- “后续事项”。
- 折叠的“原始内容 N｜首次记录/补充”。
- 记录之间的水平分隔线。

禁止输出 HTML 管理注释、Base64 数据、记录 ID、排序键和索引路径。

## 迁移现有记录

迁移必须可重复执行且默认保守失败：

1. 停止 LaunchAgent，避免迁移期间收到新写入。
2. 只读扫描现有受管理的每日文件并解析旧注释。
3. 将待迁移文件和旧结构化数据备份到 Mac 本地带时间戳的只读备份目录。
4. 为每条旧记录及其原文片段生成新的随机编号，再写入隐藏索引并重新打开验证；旧的消息派生编号只保留在 Mac 本地迁移备份中。
5. 根据索引生成不含内部注释的 Markdown 临时文件。
6. 比对记录数量、日期、排序、整理内容、后续事项和全部原文。
7. 验证通过后原子替换每日 Markdown；任一步失败则保持原文件不变。
8. 重启服务并执行健康检查。

迁移工具不得把备份、日志或飞书标识写入 U 盘。

## 便携性

U 盘移动后，Markdown 和 Skill 可以直接查看。另一台 Mac 若要恢复飞书自动处理，需要安装本地组件、配置独立凭据并指向该 U 盘；组件随后读取随盘索引并继续补充旧记录。运行状态、账号授权和日志不会随 U 盘迁移。

## 安全与冲突处理

- 只允许写入已验证 Vault 下的 `亚信工作/每日工作/` 和固定隐藏索引路径。
- 所有路径经过 `realpath` 与根目录包含性检查，拒绝符号链接逃逸。
- Markdown 与索引都使用期望内容比对和原子替换，检测并发修改。
- 若用户手工修改工作日记导致它与索引不一致，程序询问或停止写入，不使用旧索引覆盖手工内容。
- 索引损坏时不调用 AI 猜测恢复；使用本地备份或经过明确确认的导入流程。

## 测试与验收

新增自动化测试覆盖：

- Markdown 中不存在 `llw-record-*`、Base64 管理行或记录 ID。
- 索引权限、Schema 校验、原子写入和损坏拒绝。
- 新建、排序、补充原日期和重复消息不重复入库。
- 本地写入预留在重启和重复投递后复用同一组随机编号。
- 索引先提交而 Markdown 失败后的安全恢复。
- 旧格式迁移后记录数量、所有原文和整理内容完全一致。
- U 盘索引可以由另一份本地组件读取。
- 用户手工修改冲突不会被静默覆盖。
- 当前完整测试套件继续通过。

真实迁移验收时还需确认：现有工作日记源码不再出现内部编码、Obsidian 阅读正常、飞书发送一条新记录和一条自然语言补充均写入正确日期，并且 U 盘内不存在凭据、飞书身份或消息标识。
